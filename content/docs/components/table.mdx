## Props

The examples above cover the most common ways to use `Table`. Start with the base layout, then layer in sorting, sticky headers, or custom cell rendering as your dataset grows more complex.

| Prop | Type | Default | Description |
| -------------- | ------------------------- | ----------- | ------------------------------- |
| `columns` | `TableColumn<T>[]` | - | Column definitions used to render the header and cells |
| `data` | `T[]` | - | Rows rendered in the table body |
| `caption` | `string` | - | Optional table caption |
| `stickyHeader` | `boolean` | `false` | Keeps the header fixed while scrolling |

This component also accepts the shared `theme`, `primary`, `secondary`, `overlap`, and `paper` overrides.

## Column definition

```tsx
interface TableColumn<T> {
  key: keyof T | string;
  header: string;
  width?: number | string;
  align?: "left" | "center" | "right";
  sortable?: boolean;
  render?: (value: unknown, row: T, index: number) => ReactNode;
}
```

## Design notes

Header row: solid `var(--riso-primary)` fill, white 9px Space Grotesk uppercase. Column dividers are `1px rgba(255,255,255,0.2)`.

Odd data rows use an absolutely-positioned halftone dot overlay (`radial-gradient` at 4px spacing, 7% opacity) - not a background color tint.

Row hover: `color-mix(in srgb, var(--riso-secondary) 12%, transparent)` - a very light secondary tint, not grey.

Sort: active column shows bright white chevrons; inactive sortable columns show `rgba(255,255,255,0.5)` chevrons. Both up/down arrows always visible - active direction highlighted.

That restraint is what keeps `Table` feeling printed instead of purely digital: the offsets stay small, the edges stay sharp, and the ink contrast does most of the visual work.
